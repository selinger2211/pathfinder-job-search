# Pathfinder Domain Model and Glossary

## 1. Purpose

This document is the canonical reference for domain terminology in Pathfinder. It resolves ambiguities in how terms like "positioning," "tier," "stage," and "enrichment" are used across modules, ensuring consistent implementation and communication.

## 2. Core Entities

### Role
A specific job opportunity being tracked. Identified by `id` (format: `role-{timestamp}`). Belongs to exactly one Company. Has one lifecycle stage and one priority tier. Represents a single position that the user is actively or passively tracking through their job search journey.

### Company
An employer being tracked. Identified by `name` (string, NO numeric ID). Can have multiple Roles. Carries enrichment metadata including domain, funding, headcount, and mission. The company record is shared across all roles at that employer.

### Connection
A person in the user's professional network. Identified by `id`. Can be linked to multiple Roles. Tracks communication history, contact information, and relationship metadata. Created and managed through the Pipeline.

### LinkedIn Connection
An imported 1st-degree LinkedIn connection. Stored in `pf_linkedin_network`. NOT the same as a tracked Connection — LinkedIn connections become tracked Connections only when the user explicitly clicks "+ Track." These are raw imports from the LinkedIn sync.

## 3. Ambiguous Terms Resolved

### "Positioning" — 3 Different Meanings

| Context | Meaning | Values | Field |
|---------|---------|--------|-------|
| Pipeline/Role | Career path type | "ic" (individual contributor) or "management" | `role.positioning` |
| Comp Intelligence | Seniority level | "senior", "staff", "principal", "director", "VP" | `role.targetLevel` |
| Resume Builder | Value proposition angle | "problem solver", "people builder", "technical leader", etc. | Resume generation parameter (not stored) |

**Recommendation:** Rename to avoid ambiguity:
- `role.careerPath` for IC/management distinction
- `role.level` for seniority classification
- Keep resume positioning as a generation-time parameter with no persistent storage

### "Tier" — 2 Meanings

| Context | Meaning | Values |
|---------|---------|--------|
| Company/Role priority | User's interest level | "hot", "active", "watching" |
| Feed scoring | Score-based auto-classification | Score ≥80 = hot, 60-79 = active, <60 = watching |

**Relationship:** These are the SAME concept applied at different points. The Feed module auto-assigns tier based on score; the user can manually override the tier in the Pipeline. Think of Feed tier as "system-assigned" and Pipeline tier as "user-assigned."

### "Stage" — Lifecycle Stage (Canonical Definition)

Always means the same thing: the current position in the role's lifecycle. There are exactly 8 values:

```
discovered → researching → outreach → applied → screen → interviewing → offer → closed
```

**Note:** The "closed" stage covers both accepted offers AND rejections. To determine the outcome, check the last entry in `role.stageHistory` for metadata indicating whether it was an offer acceptance or rejection.

### "Enrichment"

The process of fetching additional data about a company or role from external sources. Enrichment is always asynchronous and optional.

- **JD Enrichment:** Fetching the full job description text from LinkedIn, ATS APIs, web search, or other job boards
- **Company Enrichment:** Fetching company metadata (funding, headcount, mission, tech stack, recent news) from Clay, web scraping, or similar sources
- **Connection Enrichment:** Finding mutual connections at a target company or identifying relevant connection paths

### "Brief" vs "Research Brief"

- **Research Brief** = the complete 14-section output generated by the Research Brief module
- **Brief Section** = one of the 14 individual sections within a Research Brief
- **Convention:** Never use just "brief" without the "Research" prefix in code comments, documentation, or module communication. Always write "Research Brief" to avoid confusion with other brief-like outputs.

### "Nudge"

A contextual suggestion from the Dashboard's rule engine. NOT a notification or alert. Nudges appear exclusively on the Dashboard page — they are embedded UI elements and never push to external channels or sidebar notifications. Nudges are dismissible and context-aware.

### "Artifact"

A file attached to a role: research brief PDF, resume PDF/DOCX, uploaded document, or other supporting files. Stored in IndexedDB or MCP filesystem. Individual artifacts are tracked in `role.artifacts[]` with metadata (type, filename, creation date, reference).

## 4. Data Ownership

This table defines the single source of truth for each data entity. Modules must read from the owner and never maintain local copies that diverge.

| Data | Owner Module | Storage Key | Who Reads |
|------|-------------|-------------|-----------|
| Roles (lifecycle, stage, metadata) | Pipeline | `pf_roles` | All modules |
| Companies (profiles, enrichment) | Pipeline | `pf_companies` | All modules |
| Connections (tracked) | Pipeline | `pf_connections` | Dashboard, Outreach, Research |
| LinkedIn Network | Sync Hub (import) | `pf_linkedin_network` | Pipeline, Dashboard, Feed |
| Feed Queue (scored roles) | Job Feed | `pf_feed_queue` | Dashboard |
| Preferences (scoring weights) | Job Feed | `pf_preferences` | Feed only |
| Salary Preferences | Job Feed | `pf_salary_prefs` | Feed, Pipeline, Comp |
| Debriefs | Debrief | `pf_debriefs` | Dashboard, Research |
| Mock Sessions | Mock Interview | `pf_mock_sessions` | Dashboard |
| Outreach Messages | Outreach | `pf_outreach_messages` | Dashboard |
| Calendar Events | Calendar | `pf_calendar_events` | Dashboard, Pipeline |
| Comp Data | Comp Intelligence | `pf_comp_data` | Pipeline, Dashboard |
| API Key | Settings (any module) | `pf_anthropic_key` | All Claude-calling modules |
| Theme | Any module | `pf_theme` | All modules (UI only) |

## 5. State Machine: Role Lifecycle

### Lifecycle Diagram

```
discovered → researching → outreach → applied → screen → interviewing → offer → closed
     ↓           ↓           ↓         ↓         ↓          ↓           ↓
   (reject)   (reject)   (reject)  (reject)  (reject)   (reject)   (accept/reject)
```

### Transition Rules

- **Forward Transitions:** The standard happy path progresses left-to-right through the stages
- **Rejection at Any Stage:** A role can transition directly to `closed` from any stage if rejected or withdrawn by the user
- **Offer Resolution:** Transition from `offer` → `closed` can result in either acceptance or rejection; check `stageHistory[last].metadata.outcome` to determine which
- **Backward Transitions:** Allowed (e.g., `interviewing` → `screen` if an additional screening round is added after an interview)

### Implementation Notes

- Each stage transition should update `role.stageHistory` with timestamp and transition metadata
- The `closed` stage is terminal — no further transitions are possible
- User actions (rejection, withdrawal) should always be timestamped and logged in history for audit purposes
