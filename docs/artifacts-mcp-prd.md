## 1. Purpose

The Artifacts MCP Server is the shared file system for Pathfinder. It provides a single, structured, queryable repository for every document the system produces or the user uploads: research briefs, resumes, job descriptions, fit assessments, debriefs, mock interview transcripts, outreach drafts, cover letters, homework submissions, and offer letters.

**Without the Artifacts MCP Server:** Documents scatter. Research briefs might live in browser localStorage (cached but unportable). Resumes end up in Downloads. Job descriptions are pasted into various agents but never stored. Debriefs are lost after the session ends. There is no way to search across the corpus, no versioning, no structure. Each agent operates in isolation, reinventing retrieval and storage.

**With the Artifacts MCP Server:** Any Claude-powered agent in the system can call standard MCP tools to save a document, retrieve it later, list documents with flexible filtering, search across all content, tag artifacts dynamically, and delete or archive artifacts. The browser has no direct access to the filesystem — the MCP server is the single source of truth. This enforces privacy (API key never leaves the browser), enables rich querying (find all resumes for company X, all debriefs after date Y), and allows agents to compose on top of prior work.

### Why MCP?

Without MCP, the Artifacts layer would require one of three architectures:

1. **localStorage only.** Fast, but can't exceed ~5-10MB, limited to the browser tab that wrote it, and no filesystem for DOCX/PDF binaries.
2. **Cloud backend.** Requires authentication, privacy trade-offs, and complexity. Pathfinder avoids this by design.
3. **Scattered files.** Users manage their own file storage (~/Downloads, ~/Documents, etc.). No structure, no tagging, no search.

MCP solves this: it's a local, stdin/stdout protocol that runs on the user's machine. The server can:
- Write DOCX, PDF, JSON, Markdown, HTML files to the filesystem
- Index metadata (filename, tags, artifact type, date, related roles/companies)
- Return blazing-fast metadata queries without reading full file contents
- Enforce privacy (no API keys in the browser layer)
- Support any tool that speaks MCP (Claude, any other Claude-powered agent, future tools)

The server runs on localhost and is invoked by browser agents via HTTP bridge or directly via MCP protocol if the agent is a Claude skill/workflow.

### Design Principles

1. **Single source of truth for files.** All artifacts generated by Pathfinder agents, plus user uploads, live in `~/.pathfinder/artifacts/`. No sync conflicts, no version divergence across tabs.

2. **Metadata-driven retrieval.** Every artifact has structured metadata: type, tags, related role ID, related company, created date, modified date, source agent. Queries leverage this index for sub-millisecond lookups even with 1000+ artifacts.

3. **MCP protocol is universal.** Any agent that speaks MCP can read/write artifacts. Not locked to a single framework or transport. Extensible to future tools.

4. **Local storage with no sync.** No cloud backend. No incremental sync logic. No merge conflicts. The filesystem is the database. Archiving (soft delete) preserves audit trails without consuming active disk space.

5. **Privacy by default.** The browser layer contains no file paths, no metadata index, and no API keys. All queries flow through the MCP server running on the user's machine. If the user wants to back up artifacts, they copy the `~/.pathfinder/artifacts/` directory — no proprietary export formats.

## 2. Architecture

### 2.1 System Layers

**MCP Server** (Node.js, runs on localhost:3457 or similar)
- Exposes nine tools: `save_artifact`, `get_artifact`, `list_artifacts`, `search_artifacts`, `tag_artifact`, `delete_artifact`, `generate_brief_section`, `backup_pipeline`, `restore_pipeline`
- Manages metadata index in `~/.pathfinder/artifacts/.index.json`
- Writes artifact files to `~/.pathfinder/artifacts/{type}/{artifactId}/`
- Handles lifecycle: creating, retrieving, tagging, archiving

**HTTP Bridge** (optional, for browser agents)
- If agents run in the browser (not as Claude skills), an HTTP bridge on localhost:3456 translates requests to MCP calls
- Example: browser calls `POST /artifacts/save` → bridge calls MCP `save_artifact` → returns `{ artifactId, path }`
- Bridge keeps API keys server-side; browser never holds them

**Browser Integration Layer** (lightweight, read-mostly)
- Agents call bridge endpoints or MCP tools
- Results are cached in localStorage for instant UI feedback
- Any write operation (save, delete, tag) invalidates the relevant cache keys
- No file I/O in the browser; all calls are remote

**Research Brief Generator** (currently resident in this module, see 2.5)
- Claude.ts with 14 section-specific prompts
- Runs on localhost:3456 (HTTP bridge)
- Generates sections server-side, caches each, saves final brief via `save_artifact`

## 3. Data Model

### 3.1 Artifact Schema

Every artifact has two parts: **content** and **metadata**.

**Metadata** (stored in `metadata.json`, also indexed centrally)

```json
{
  "artifactId": "string (UUID)",
  "type": "enum (research_brief|resume|jd_snapshot|fit_assessment|debrief|mock_interview|outreach_draft|cover_letter|homework_submission|offer_letter)",
  "filename": "string",
  "tags": ["string"],
  "company": "string (optional, canonical company name or Pathfinder ID)",
  "roleId": "string (optional, reference to pipeline role)",
  "createdAt": "ISO 8601 timestamp",
  "modifiedAt": "ISO 8601 timestamp",
  "sourceAgent": "string (research-brief-agent|resume-tailor|debrief-agent|...)",
  "sizeBytes": "number",
  "excerpt": "string (first 200 chars of content or summary)",
  "archived": "boolean (default false)",
  "archivedAt": "ISO 8601 timestamp (optional)",
  "checksum": "string (SHA-256 of content, for dedup and integrity)"
}
```

**Content**

Format depends on type:
- `research_brief`, `outreach_draft`, `debrief`: UTF-8 text (HTML, Markdown)
- `resume`, `cover_letter`, `offer_letter`: Binary (DOCX, PDF)
- `fit_assessment`, `mock_interview`: JSON
- `jd_snapshot`: UTF-8 text
- `homework_submission`: Any format (MIME type stored in metadata)

## 4. API / Tools Specification

### Overview

The MCP server exposes six tools. All are async, return structured JSON, and fail safely (never corrupt the index).

---

### 4.1 `save_artifact`

**Purpose:** Save a new artifact or overwrite an existing one.

**Input Schema**

```json
{
  "type": "string (enum: research_brief|resume|jd_snapshot|...)",
  "content": "string (if text) or Buffer (if binary)",
  "filename": "string (e.g., 'Stripe-Product-Manager.html')",
  "tags": ["string"],
  "company": "string (optional)",
  "roleId": "string (optional, reference to pipeline role)",
  "sourceAgent": "string (e.g., 'research-brief-agent')",
  "metadata": "object (optional, extra fields)"
}
```

**Output Schema**

```json
{
  "artifactId": "string",
  "path": "string (absolute filesystem path)",
  "filename": "string",
  "sizeBytes": "number",
  "created": "boolean (true if new, false if overwrote)",
  "createdAt": "ISO 8601 timestamp",
  "modifiedAt": "ISO 8601 timestamp"
}
```

**Behavior**

- Generates a UUID `artifactId` if not provided
- Creates directory `~/.pathfinder/artifacts/{type}/{artifactId}/`
- Writes content to `~/.pathfinder/artifacts/{type}/{artifactId}/content.{ext}`
- Writes metadata to `~/.pathfinder/artifacts/{type}/{artifactId}/metadata.json`
- Updates central index (`.index.json`)
- Computes checksum for deduplication
- Returns full artifact metadata including path (for browser to fetch if needed)

**Example (Research Brief Agent saving a brief)**

```json
// Request
{
  "type": "research_brief",
  "content": "<html>...</html>",
  "filename": "Stripe-Product-Manager.html",
  "tags": ["stripe", "payment-infrastructure", "prod-strategy"],
  "company": "stripe",
  "roleId": "role-789",
  "sourceAgent": "research-brief-agent"
}

// Response
{
  "artifactId": "abc-123-def-456",
  "path": "/Users/user/.pathfinder/artifacts/research_brief/abc-123-def-456/content.html",
  "filename": "Stripe-Product-Manager.html",
  "sizeBytes": 45672,
  "created": true,
  "createdAt": "2026-03-10T14:22:00Z",
  "modifiedAt": "2026-03-10T14:22:00Z"
}
```

### 4.2 `get_artifact`

**Purpose:** Retrieve a single artifact by ID.

**Input Schema**

```json
{
  "artifactId": "string",
  "includeContent": "boolean (default true; if false, return metadata only)"
}
```

**Output Schema**

```json
{
  "artifactId": "string",
  "type": "string",
  "filename": "string",
  "tags": ["string"],
  "company": "string (optional)",
  "roleId": "string (optional)",
  "createdAt": "ISO 8601 timestamp",
  "modifiedAt": "ISO 8601 timestamp",
  "sourceAgent": "string",
  "sizeBytes": "number",
  "excerpt": "string",
  "archived": "boolean",
  "content": "string or Buffer (if includeContent=true and not archived)"
}
```

**Behavior**

- Looks up artifact in index by ID
- Returns metadata + content if requested
- If artifact is archived, returns metadata only; `includeContent` is ignored
- Fails with 404 if artifact doesn't exist
- Content is streamed for large files (DOCX, PDFs)

**Example**

```json
// Request
{ "artifactId": "abc-123-def-456" }

// Response
{
  "artifactId": "abc-123-def-456",
  "type": "research_brief",
  "filename": "Stripe-Product-Manager.html",
  "tags": ["stripe", "payment-infrastructure"],
  "company": "stripe",
  "roleId": "role-789",
  "createdAt": "2026-03-10T14:22:00Z",
  "modifiedAt": "2026-03-10T14:22:00Z",
  "sourceAgent": "research-brief-agent",
  "sizeBytes": 45672,
  "excerpt": "Stripe is a payments infrastructure company...",
  "archived": false,
  "content": "<html>...</html>"
}
```

### 4.3 `list_artifacts`

**Purpose:** Retrieve all artifacts matching optional filters.

**Input Schema**

```json
{
  "type": "string (optional, enum: research_brief|resume|...)",
  "tags": ["string (optional, AND logic)"],
  "company": "string (optional)",
  "roleId": "string (optional)",
  "dateRange": {
    "startDate": "ISO 8601 date or timestamp (optional)",
    "endDate": "ISO 8601 date or timestamp (optional)"
  },
  "includeArchived": "boolean (default false)",
  "limit": "number (default 100, max 1000)",
  "offset": "number (default 0)"
}
```

**Output Schema**

```json
{
  "artifacts": [
    {
      "artifactId": "string",
      "type": "string",
      "filename": "string",
      "tags": ["string"],
      "company": "string",
      "roleId": "string",
      "createdAt": "ISO 8601 timestamp",
      "modifiedAt": "ISO 8601 timestamp",
      "sourceAgent": "string",
      "sizeBytes": "number",
      "excerpt": "string",
      "archived": "boolean"
    }
  ],
  "total": "number",
  "limit": "number",
  "offset": "number",
  "hasMore": "boolean"
}
```

**Behavior**

- Queries central index with all filters
- Returns metadata only (no content)
- Supports pagination via `limit` and `offset`
- Tag filters use AND logic (all tags must match)
- Date range is inclusive on both ends
- Sorted by `modifiedAt` descending (most recent first)

**Example (Get all resumes for a specific role)**

```json
// Request
{
  "type": "resume",
  "roleId": "role-789",
  "limit": 10
}

// Response
{
  "artifacts": [
    {
      "artifactId": "resume-001",
      "type": "resume",
      "filename": "Stripe-PM-v2.docx",
      "tags": ["stripe", "tailored"],
      "company": "stripe",
      "roleId": "role-789",
      "createdAt": "2026-03-10T12:00:00Z",
      "modifiedAt": "2026-03-10T14:15:00Z",
      "sourceAgent": "resume-tailor",
      "sizeBytes": 28500,
      "excerpt": "...",
      "archived": false
    }
  ],
  "total": 1,
  "limit": 10,
  "offset": 0,
  "hasMore": false
}
```

### 4.4 `search_artifacts`

**Purpose:** Full-text search across artifact content and metadata.

**Input Schema**

```json
{
  "query": "string (search terms)",
  "type": "string (optional, filter by type)",
  "company": "string (optional)",
  "limit": "number (default 20, max 100)",
  "offset": "number (default 0)"
}
```

**Output Schema**

```json
{
  "results": [
    {
      "artifactId": "string",
      "type": "string",
      "filename": "string",
      "tags": ["string"],
      "company": "string",
      "roleId": "string",
      "createdAt": "ISO 8601 timestamp",
      "relevance": "number (0-1, higher = better match)",
      "matchedSnippet": "string (context around match, max 200 chars)"
    }
  ],
  "total": "number",
  "limit": "number",
  "offset": "number",
  "hasMore": "boolean",
  "query": "string",
  "executionTimeMs": "number"
}
```

**Behavior**

- Searches filename, tags, excerpt, and full content (for text types)
- Returns results sorted by relevance (TF-IDF or simple token matching)
- For binary types (DOCX, PDF), searches metadata only
- Execution time is tracked (useful for performance monitoring)
- Returns matched context snippets to help user verify results

### 4.5 `tag_artifact`

**Purpose:** Add or remove tags from an existing artifact.

**Input Schema**

```json
{
  "artifactId": "string",
  "addTags": ["string (optional)"],
  "removeTags": ["string (optional)"]
}
```

**Output Schema**

```json
{
  "artifactId": "string",
  "tags": ["string"],
  "modifiedAt": "ISO 8601 timestamp"
}
```

**Behavior**

- Updates metadata.json and central index
- `addTags` are merged (no duplicates)
- `removeTags` are removed (no-op if tag doesn't exist)
- At least one of `addTags` or `removeTags` must be provided
- Returns updated tag list and modification timestamp

**Example (Add "final-round" tag to a debrief)**

```json
// Request
{
  "artifactId": "debrief-001",
  "addTags": ["final-round", "stripe-pm-role"]
}

// Response
{
  "artifactId": "debrief-001",
  "tags": ["stripe", "final-round", "stripe-pm-role"],
  "modifiedAt": "2026-03-10T16:45:00Z"
}
```

### 4.6 `delete_artifact`

**Purpose:** Delete (soft-archive) an artifact.

**Input Schema**

```json
{
  "artifactId": "string",
  "permanent": "boolean (default false; true = hard delete)"
}
```

**Output Schema**

```json
{
  "artifactId": "string",
  "deleted": "boolean",
  "archived": "boolean (true if soft-deleted, false if hard-deleted)",
  "archivedAt": "ISO 8601 timestamp (if soft-deleted)"
}
```

**Behavior**

- Default: soft delete (move to `.archive/`, mark as archived, keep in index with `archived=true`)
- If `permanent=true`: hard delete (remove from filesystem and index)
- Soft delete is reversible (can be unarchived later)
- Hard delete is irreversible; used only for true cleanup
- Returns confirmation with archive status

**Example (Soft-delete a failed draft)**

```json
// Request
{ "artifactId": "outreach-draft-001" }

// Response
{
  "artifactId": "outreach-draft-001",
  "deleted": true,
  "archived": true,
  "archivedAt": "2026-03-10T17:00:00Z"
}
```

## 6. Data Flow & Integration

### 6.1 How Agents Interact with Artifacts MCP

**Research Brief Agent**

```
1. Receives: JD text, company profile, user resume
2. Generates: 14 section prompts, streams each via Claude
3. Calls: save_artifact({
     type: "research_brief",
     content: htmlContent,
     filename: "Stripe-Product-Manager.html",
     tags: ["stripe", "product-manager", "payment"],
     company: "stripe",
     roleId: "role-789",
     sourceAgent: "research-brief-agent"
   })
4. Gets: artifactId, path
5. Returns: artifactId to browser (browser caches in localStorage)
6. Browser: fetches artifact content from path or via get_artifact
```

**Resume Tailor Agent**

```
1. Receives: JD, fit assessment, positioning statement
2. Generates: DOCX resume
3. Calls: save_artifact({
     type: "resume",
     content: docxBuffer,
     filename: "Stripe-PM-Tailored.docx",
     tags: ["stripe", "tailored"],
     company: "stripe",
     roleId: "role-789",
     sourceAgent: "resume-tailor"
   })
4. Gets: artifactId, path
5. Browser: displays download link, caches in localStorage
6. User: can retrieve this resume later via list_artifacts({roleId, type: "resume"})
```

**Post-Interview Debrief Agent**

```
1. Receives: interview notes, company/role context
2. Generates: structured debrief (Markdown)
3. Calls: save_artifact({
     type: "debrief",
     content: markdownContent,
     filename: "Stripe-PM-Screen-Debrief.md",
     tags: ["stripe", "screen-round", "2026-03-10"],
     company: "stripe",
     roleId: "role-789",
     sourceAgent: "debrief-agent"
   })
4. Also calls: tag_artifact(artifactId, { addTags: ["reviewed", "actionable"] })
5. Returns: artifactId to browser
6. Dashboard: queries list_artifacts({ company: "stripe", type: "debrief" }) to show interview history
```

**User Upload (e.g., offer letter)**

```
1. User: uploads PDF via Pipeline Tracker UI
2. Browser: calls save_artifact({
     type: "offer_letter",
     content: pdfBuffer,
     filename: "Stripe-Offer.pdf",
     tags: ["stripe", "offer"],
     company: "stripe",
     roleId: "role-789",
     sourceAgent: "pipeline-tracker"
   })
3. Gets: artifactId
4. Browser: stores artifactId in pipeline role record (localStorage)
5. User: can view/download via role detail page
```

### 6.2 Query Patterns

**Pipeline Tracker: "Show all artifacts for this role"**

```json
list_artifacts({ roleId: "role-789" })
```

## 8. Security & Privacy

### 8.1 Privacy Model

**No cloud sync.** All artifacts live on the user's machine in `~/.pathfinder/artifacts/`. They are never uploaded to a server.

**API key security.** The Claude API key is stored server-side (in the Node.js MCP server process, or in a browser-side environment variable loaded at startup). The browser layer does not have direct access to the key. All generation requests (e.g., research brief) flow through the HTTP bridge or MCP server.

**No account/authentication.** There are no user accounts, logins, or third-party authentication. The system is single-user and single-machine.

### 8.2 Data Integrity

**Index validation.** On startup, the MCP server validates the central index against the filesystem. If index entries refer to missing files, they are removed. If files exist without index entries, they are indexed.

**Checksums.** Every artifact stores a SHA-256 checksum in metadata. On retrieval, the checksum is verified to detect corruption.

**Soft deletes.** Artifacts are soft-deleted (moved to `.archive/`) by default, preserving audit trails. Hard deletes require explicit `permanent=true` flag.

### 8.3 Permissions & Access

**Single-user system.** No role-based access control. The user who owns the system owns all artifacts.

**Filesystem permissions.** The `~/.pathfinder/artifacts/` directory is created with mode `0700` (user read/write/execute only). This prevents other users on the same machine from accessing job search data.

## 9. Performance & Scalability

### 9.1 Index Design

**In-memory index.** The central `.index.json` is loaded into memory on server startup. For up to 1000 artifacts, this is negligible (~5MB).

**Query performance.** All queries (list, search, tag) operate on the in-memory index:
- `list_artifacts` with filters: <10ms
- `search_artifacts` with full-text: <100ms (TF-IDF scoring)
- `get_artifact` (fetch by ID): <1ms

**For >1000 artifacts.** If the system grows beyond 1000 artifacts, replace in-memory index with a lightweight embedded DB (SQLite, leveldb) to maintain sub-100ms query times.

### 9.2 Storage

**Typical usage:** A user running a 6-month job search with 50 active roles might generate:
- 50 research briefs (~50KB each) = 2.5MB
- 100 tailored resumes (~50KB each) = 5MB
- 200 debriefs (~10KB each) = 2MB
- 100 mock interview transcripts (~20KB each) = 2MB
- Miscellaneous (JD snapshots, cover letters, offers) = ~2MB
- **Total: ~13.5MB**

This is negligible on modern machines. No storage concerns for single-user, single-machine system.

### 9.3 Streaming

For large files (PDFs, high-fidelity DOCXs >10MB), implement streaming in the HTTP bridge:
- `/artifacts/get/{id}` with `Range` header support
- Stream content in chunks to avoid loading entire file into memory

## Conclusion

The Artifacts MCP Server is the central file system for Pathfinder, enabling:

- **No scattered files.** Every document is stored, indexed, and queryable in one place.
- **Stateful composition.** Agents build on prior work (resumes, briefs, debriefs) without re-entering data.
- **Privacy-first design.** No cloud, no account, no API key exposure.
- **Extensibility.** Future agents (and even external tools) can interact via standard MCP tools.
- **Simple architecture.** Filesystem + metadata index + HTTP bridge. No complex databases or sync logic.

By treating artifacts as first-class entities in the Pathfinder system, we solve the fragmentation problem that plagues existing job search tools and enable a true agentic architecture where information flows seamlessly between specialized agents.
