**Version:** v2.0.0 (overhaul)
**Last Updated:** 2026-03-10
**Status:** Draft — pending approval

---

## 1. Purpose

The Research Brief is the preparation engine of Pathfinder. Before every interview loop, networking call, or application decision, you should be able to open a brief and know: what this role actually needs, what this company is doing right now, who you'd work with, how your experience maps, and exactly what to say.

**v1 was a UI shell with template strings.** v2 is a ground-up rebuild where every section is generated by Claude via MCP tool calls, fed with real data (JD text, company profile, your resume bullets, your connections), and sourced with citations per Section 7.12 of the main PRD.

### Design Principles

1. **JD is the anchor.** Every section reads the job description. Nothing generates in a vacuum. If there's no JD, the brief says so and asks you to add one.
2. **Your experience is the lens.** The brief isn't a Wikipedia article about the company — it's a preparation document for *you* interviewing for *this role*. Your resume bullets, story bank, and connections feed into every relevant section.
3. **Trust but Verify.** Every claim is cited. Every source is timestamped. Stale data is flagged. The user can click through to the original source. (See main PRD Section 7.12.)
4. **MCP-generated, browser-displayed.** Generation happens via MCP tool calls (Claude processes the prompts server-side). The browser module is a read-only renderer that fetches completed sections from the MCP server and displays them. No API keys in the browser.

## 2. Architecture

### Generation Flow

```
User clicks "Generate Brief" in browser
        │
        ▼
Browser writes request to localStorage
  pf_brief_request = { roleId, companyId, sections: [1..13], timestamp }
        │
        ▼
MCP Tool: pf_generate_brief_section
  (called once per section, can run in parallel batches)
        │
        ├── Reads from localStorage: pf_roles, pf_companies, pf_connections,
        │   pf_bullet_bank, pf_story_bank, pf_comp_cache
        │
        ├── Reads from MCP artifacts: existing research briefs, JD snapshots
        │
        ├── Claude generates section content with inline citations
        │
        ├── Saves citations via pf_save_citation
        │
        └── Saves section content via pf_save_artifact (type: research_brief)
                │
                ▼
Browser polls/reads completed sections from MCP artifacts
  and renders them with citation markers
```

### Data Inputs Per Section

Every section receives a **base context** containing:

- **JD text** — the full job description from `role.jdText`
- **Role metadata** — title, level, positioning (IC/management), stage, URL, dateAdded
- **Company profile** — name, domain, fundingStage, headcount, missionStatement, companyType, remotePolicy, techStack

Sections that need additional data specify it below. If a required input is missing, the section generates with a visible callout: "⚠ This section would be stronger with [missing data]. Add it in Pipeline → Role Detail."

## 3. Section Specifications

### Section 1: Role Decode

**Purpose:** Parse the JD and surface what this role actually needs — the stuff between the lines that a recruiter won't say.

**Inputs:** Base context (JD is critical)

**Output structure:**
- **The Real Problem** — One paragraph: what business problem is this hire solving? Inferred from JD language, team description, and company stage.
- **Explicit Requirements** — Bulleted list of what the JD directly asks for (years of experience, specific skills, domain knowledge). Each tagged as `must-have` or `nice-to-have` based on JD language ("required" vs "preferred", "bonus", "ideally").
- **Implied Requirements** — What the JD signals without stating. Examples: "comfortable with ambiguity" → early-stage product, undefined roadmap. "Cross-functional leadership" → no direct reports, influence-based role. "0-to-1" → greenfield, expect to define the problem.
- **Level Signals** — What seniority indicators are in the JD? Scope of ownership, decision-making authority, team size mentioned, reporting line hints. How does this match the stated title?
- **Red Flags & Watch Items** — Anything unusual: unrealistic scope for the level, conflicting requirements, buzzword density suggesting the role isn't well-defined yet.

**Prompt guidance:** The model should read the JD as a hiring manager would write it — with constraints, urgency signals, and organizational context embedded in word choices. This is not a summary; it's an interpretation.

**Citations:** Each requirement traced to the specific JD line(s) it comes from. Source type: `job_board` if imported from URL, `manual_entry` if pasted.

### Section 2: Company Now

**Purpose:** What's happening at this company *right now* that's relevant to *this role*. Not a founding story — a situational briefing.

**Inputs:** Base context + `company.recentNews` (if populated) + web enrichment data

**Output structure:**
- **Current Moment** — 2-3 paragraphs on what the company is doing this quarter. Product launches, strategic shifts, market moves, leadership changes. Everything timestamped.
- **Why This Role, Why Now** — Hypothesis: what's driving this hire? New product line? Team scaling? Backfill? Derived from company context + JD language.
- **Key Metrics** (if available) — Revenue signals, growth rate, user base, market share. For public companies: recent earnings highlights. For private: last known fundraise, headcount trajectory.
- **Leadership** — CEO, relevant VP/SVP over this function, CTO (if engineering-adjacent role). One line each: name, tenure, background, notable decisions.

**Prompt guidance:** Prioritize recency. A company overview from 2024 is stale. If the model's training data is limited, it should say so and flag sections for manual research. Never fabricate news or metrics.

**Citations:** Every fact sourced. News articles, earnings calls, press releases, LinkedIn profiles. Source type: `enrichment_web` with URLs and fetch dates.

### Section 3: Funding & Corporate Structure

**Purpose:** Follow the money. Who funds this company, what's the cap table story, and what's the corporate family tree.

**Inputs:** Base context + `company.fundingStage` + web enrichment data

**Output structure:**
- **Funding History** — Table: Round, Date, Amount, Lead Investor(s), Valuation (if known). Most recent round first.
- **Notable Investors & Board Members** — Names, firms, and why they matter (e.g., "Sequoia led Series B — their portfolio includes [relevant competitors]"). Board composition signals (heavy on operators vs. financial investors).
- **Corporate Structure** — Parent company (if subsidiary), subsidiaries, recent acquisitions, joint ventures. For conglomerates: where does this business unit sit?
- **Financial Health Signals** — For public companies: market cap, P/E, revenue trend. For private: runway signals (last raise date vs. headcount growth), any known profitability milestones. For both: any recent layoffs, hiring freezes, or aggressive hiring signals.
- **What This Means for You** — One paragraph connecting the financial picture to the role. Example: "Series D with 18 months since last raise and aggressive hiring suggests strong revenue growth or an upcoming raise. Equity could be valuable."

**Prompt guidance:** Use known funding databases (Crunchbase, PitchBook-style data). If the model doesn't have specific data, it should say "Not found in available data — verify on Crunchbase" rather than guessing. Never fabricate funding amounts or valuations.

**Citations:** Each funding round sourced. Investor names sourced. Source type: `enrichment_web`.

---

### Section 4: Competitive Landscape

**Purpose:** Who is this company fighting, specifically in the product area this role touches.

**Inputs:** Base context (JD text is critical for scoping the product area)

**Output structure:**
- **Direct Competitors** — 3-5 companies competing in the same product space as this role. For each: one-line description, estimated scale, key differentiator vs. the target company.
- **Indirect / Adjacent Competitors** — 2-3 companies that compete tangentially or could expand into this space.
- **Competitive Dynamics** — What's the market narrative? Is it consolidating? Land-grab phase? Feature parity race? Who's winning and why?
- **Target Company's Moat** — What's their defensible advantage? Network effects, data, distribution, brand, switching costs, regulatory?
- **Interview Leverage** — How to reference the competitive landscape in interviews without sounding like you're reading a report. One example talking point.

**Prompt guidance:** Scope competitors to the *product area the role sits in*, not the company overall. A PM role on Stripe's billing product competes with Chargebee and Recurly, not with Square's POS hardware. The JD should provide scoping signals.

**Citations:** Competitor claims sourced. Market data sourced. Source type: `enrichment_web` or `ai_generated` (with explicit "based on model knowledge as of [date]" flag).

### Section 5: Team & Org Intelligence

**Purpose:** Who would you work with? Who's the hiring manager? What does the org look like?

**Inputs:** Base context + `role.interviewerNames` (if provided) + `pf_connections` (for cross-reference)

**Output structure:**
- **Likely Reporting Line** — Inferred from JD: who does this role report to? VP Product? Director? Group PM? What does that imply about scope and visibility?
- **Team Shape** — Estimated team size, composition (PMs, engineers, designers, data), and org model (squad, pod, functional) — inferred from JD language.
- **Adjacent Teams** — Who would this role interact with most? Sales, marketing, engineering leadership, customer success? Based on JD's "cross-functional" mentions.
- **Interviewer Profiles** (if names provided) — For each interviewer: name, likely title (from LinkedIn if known), estimated tenure, background summary, what they probably care about in this interview. If interviewer names are not provided, show a prompt: "Add interviewer names in the sidebar to unlock this section."
- **Org Observations** — Any signals about the org from the JD or public info: new team (no established process), re-org (role reporting line changed), rapid growth (junior team needs senior leadership).

**Prompt guidance:** Interviewer research should be cautious — use only publicly available professional information (LinkedIn, company team pages, conference talks). Never fabricate personal details. Flag uncertainty: "Based on LinkedIn profile from [date], may have changed."

**Citations:** Interviewer data sourced to LinkedIn profiles or company pages. Org structure claims sourced to JD lines.

### Section 6: Network & Connections

**Purpose:** Who do you already know at this company? Who can make an intro? Turn your network into an advantage.

**Inputs:** Base context + `pf_connections` (full connections list) + `pf_roles` (for company match)

**Output structure:**
- **Direct Connections** — People in your `pf_connections` who work at this company. For each: name, title, relationship context (how you know them), last interaction date, and suggested action (warm intro request, back-channel reference, informational chat).
- **Second-Degree Paths** — Connections who previously worked there, or who are connected to known employees. "You know [Name] at [Other Company] who previously worked at [Target Company] as [Title]."
- **Suggested Outreach** — For each viable connection, a one-liner on what to say. Not a full email draft (that's the Outreach module) but a hook: "Mention your shared time at [Event/Company]. Ask about the [Team] culture."
- **No Connections?** — If no matches found: "No connections found at [Company]. Consider: LinkedIn search for [Company] + [your school/previous employers], checking if any pipeline contacts have moved there recently."

**Prompt guidance:** This section is data-driven, not generative. The model cross-references `pf_connections` against the company name. Creative suggestions for second-degree paths are welcome, but the core list must be factual from the user's data.

**Citations:** Each connection sourced to `pf_connections` data. Source type: `manual_entry` (user-entered connections).

### Section 7: Fit Analysis

**Purpose:** The honest answer to "why should they hire you?" — mapped requirement by requirement against your actual experience.

**Inputs:** Base context + `pf_bullet_bank` + `pf_resume_log` (most recent base resume) + `pf_story_bank`

**Output structure:**
- **Requirements Map** — Table with columns: Requirement (from JD), Your Evidence (matched bullet or story), Fit (green ✅ / yellow ⚠️ / red ❌), Notes.
  - **Green:** You have direct, demonstrable experience. Matched to a specific bullet or story.
  - **Yellow:** You have adjacent experience that can be positioned. Notes explain the positioning angle.
  - **Red:** Genuine gap. No relevant experience found in your data.
- **Overall Fit Score** — X/Y requirements matched (green + yellow count). Not a percentage — a concrete count so you know the ratio.
- **Gap Positioning Strategies** — For each yellow/red requirement: how to talk about it without lying. Example: "No direct fintech experience, but your payment integration work at [Previous Company] demonstrates the analytical rigor. Frame it as: 'I've built the technical intuition from the integration side and I'm eager to go deeper on the product side.'"
- **Secret Weapons** — Experience from your bullet bank that the JD doesn't ask for but would be valuable. The "and I also bring X" talking points.

**Prompt guidance:** This must be honest. The model should never fabricate matches or inflate yellow to green. A brief that tells you "you're perfect" is useless — a brief that tells you "here are your 2 gaps and here's how to talk about them" is preparation gold. If `pf_bullet_bank` is empty, flag it prominently and generate a degraded version using only JD analysis.

**Citations:** Each matched bullet sourced to `pf_bullet_bank` entry. Each requirement sourced to JD line. Source type: `manual_entry` (your bullets) + `job_board` (JD requirements).

### Section 8: Compensation Intelligence

**Purpose:** Know your number before they ask. Understand the comp structure so you negotiate from data, not guesses.

**Inputs:** Base context + `pf_comp_cache` (from Comp Intelligence module) + JD salary info (if present)

**Output structure:**
- **Expected Range** — Base salary range for this title + level + location + company stage. Source: Levels.fyi, Glassdoor, JD-stated range, or comp cache data.
- **Total Comp Structure** — How this company typically pays: base + bonus + equity. RSUs vs. options. Vesting schedule (4-year with 1-year cliff is standard; note deviations). Refresh grant norms.
- **JD Comp Signals** — Does the JD state a range? Is it broad (suggests flexibility) or tight? Is the range competitive for the market? Does the JD mention "competitive compensation" (often means below-market base, above-market equity)?
- **Negotiation Leverage Points** — Signals from the JD and company context that suggest leverage: urgency language, niche skill requirements, team growth signals, scarce talent market.
- **Your Target** — Based on the data, a suggested ask range (low / target / stretch) with rationale. Caveat: "This is a data-informed starting point, not financial advice."

**Prompt guidance:** Use levels.fyi and glassdoor-style knowledge. Never fabricate specific salary numbers — always cite the source and date. If no data available, say so and suggest where to look. Flag if the JD-stated range seems misaligned with market data.

**Citations:** Comp data sourced to levels.fyi, Glassdoor, JD text, or `pf_comp_cache`. Source type varies.

### Section 9: Strategic Challenges & First 90 Days

**Purpose:** Walk into the interview with a hypothesis. "Here's what I think you're facing and here's what I'd do about it."

**Inputs:** Base context (JD + company profile critical)

**Output structure:**
- **Top 3 Strategic Challenges** — Derived from JD priorities, company stage, and competitive landscape. For each: the challenge (one sentence), why it matters now (one paragraph), and evidence from the JD or company context.
- **Your First 90 Days** — A realistic, phase-based plan:
  - **Days 1-30:** Listen and learn. What would you try to understand? Who would you talk to? What data would you look at?
  - **Days 31-60:** Quick wins. What could you ship or unblock that demonstrates impact without requiring deep institutional knowledge?
  - **Days 61-90:** Strategic bets. What would you propose to the team/leadership? Based on which of the challenges above?
- **Talking Points** — 2-3 specific things you could say in the interview to demonstrate strategic thinking about *this* role at *this* company. Not generic frameworks — specific hypotheses.

**Prompt guidance:** The 90-day plan should be realistic for the level. An IC PM's first 90 days look different from a VP's. Avoid generic "build relationships and learn the product" — that's assumed. Focus on specifics derived from the JD and company context. It's OK to say "I'd validate whether X is true" — showing you have hypotheses is the point.

**Citations:** Each challenge sourced to JD language or company context. Source type: `job_board` + `enrichment_web`.

### Section 10: Culture & Values Decode

**Purpose:** Understand what this company *actually* values — not just what the careers page says.

**Inputs:** Base context + `company.culture` field + `company.glassdoor` (if available)

**Output structure:**
- **Stated Values** — What the company says it values (from careers page, mission statement, leadership quotes). Listed with source.
- **Signals from the JD** — What the JD language implies about culture. "Fast-paced" = long hours? "Collaborative" = consensus-driven? "Entrepreneurial" = under-resourced? Decoded, not cynically — but realistically.
- **Interview Style Signals** — What to expect: case study? System design? Behavioral? Portfolio review? Panel vs. 1:1? Inferred from company size, stage, and industry norms. If interviewer names provided, may refine based on their likely role in the loop.
- **What Gets Rewarded** — Based on company stage and JD language: is this a company that rewards shipping fast, or careful analysis? Builder or optimizer? Consensus or conviction?
- **Culture Fit Questions for You** — 3-4 questions you should ask yourself: "Am I comfortable with [X level of ambiguity]?" "Do I thrive in [consensus/conviction] cultures?" Based on the specific signals found.

**Prompt guidance:** Be nuanced. "Fast-paced" doesn't always mean burnout. "Process-oriented" doesn't always mean slow. Read the signals in context of company stage, size, and industry. Glassdoor data is noisy — weight recent reviews more heavily and note sample size.

**Citations:** Values sourced to careers page or mission statement. JD language sourced to specific lines. Glassdoor data timestamped.

### Section 11: Questions to Ask

**Purpose:** Role-specific questions organized by interview round that demonstrate you've done the homework.

**Inputs:** Base context + all previously generated sections (this section should run last or near-last)

**Output structure:**
- **Recruiter Screen** (3-4 questions) — Logistics, process, timeline, team structure validation. Things you need to know to prepare for the next round.
- **Hiring Manager** (4-5 questions) — Role-specific: what does success look like in 6 months? What's the team's biggest challenge right now? What happened to the last person in this role? Questions that demonstrate strategic thinking.
- **Technical / Panel** (3-4 questions) — Product/engineering-focused: tech stack decisions, product process, how they handle X (specific to JD requirements). Questions that demonstrate depth.
- **Executive / Final Round** (3-4 questions) — Company direction, team investment, strategic priorities. Questions that show you're thinking about the big picture.
- **Back-pocket Questions** — 2-3 questions that work in any round, if you run out or the conversation goes sideways.

**Prompt guidance:** Every question should be derivable from the brief's other sections. "I noticed in the JD that you mentioned [X] — how does that relate to [company's recent Y]?" is a good question because it proves you read the JD and know the company context. Generic questions ("tell me about the culture") should never appear here.

**Citations:** Each question linked to the brief section(s) that informed it. Source type: `ai_generated` with inputSummary listing the upstream sections.

### Section 12: TMAY Script

**Purpose:** A "Tell Me About Yourself" narrative built from your actual resume, positioned for this specific role.

**Inputs:** Base context + `pf_bullet_bank` + `pf_resume_log` (most recent) + Section 1 (Role Decode) output

**Output structure:**
- **90-Second Version** — The tight version for recruiter screens and time-constrained intros. Arc: where you've been → what you're great at → why this role.
- **2-Minute Version** — The expanded version for hiring manager and panel rounds. Same arc with more detail: specific accomplishments, named companies, quantified impact, explicit connection to the role.
- **Key Beats** — The 4-5 inflection points in your career that this script hits. For each: why it's included (maps to which JD requirement) and what to emphasize.
- **What to Skip** — Experience from your resume that you should *not* bring up unprompted for this role (irrelevant to the JD, or could raise questions without adding value).
- **Transition Line** — How to end the TMAY and hand control back: a bridge sentence that connects your story to the role and invites the next question.

**Prompt guidance:** The script should sound like a person talking, not a resume being read aloud. Use first person. Include natural transitions ("That's when I moved to..." not "Subsequently, I transitioned to..."). The 90-second version must actually be speakable in 90 seconds (~225 words). The 2-minute version ~450 words. Count them.

**Citations:** Each career beat sourced to `pf_bullet_bank` entry. Role mapping sourced to JD requirements (Section 1 output). Source type: `manual_entry` (bullets) + `job_board` (JD).

### Section 13: Likely Interview Questions

**Purpose:** Predict what they'll ask you, and have an answer ready.

**Inputs:** Base context + `pf_story_bank` + `pf_bullet_bank` + Section 1 (Role Decode) output

**Output structure:**
- **Behavioral Questions** (5-7) — Derived from JD requirements and company values. For each: the likely question, why they'd ask it (which requirement it probes), and a matched STAR story from your `pf_story_bank` (or "No matching story — consider preparing one about [topic]").
- **Technical / Product Questions** (3-5) — Domain-specific questions based on the role's product area. Case studies, estimation, system design, product strategy — whatever's appropriate for the level and function.
- **Role-Specific Curveballs** (2-3) — Questions unique to this role or company. "Why this company specifically?" "How would you handle [specific challenge from Section 9]?" "Tell me about a time you [specific JD requirement]."
- **Questions About Your Gaps** (1-3) — The questions you're dreading — the ones that probe your red/yellow items from Section 7. For each: the likely question, and a prepared answer that's honest but well-positioned.
- **Your Answer Framework** — For behavioral questions, a reminder of your STAR structure with the specific stories selected. For technical questions, your go-to frameworks.

**Prompt guidance:** Questions should be specific to this role, not generic PM/engineering interview questions. "Tell me about a time you influenced without authority" is good *only if* the JD signals cross-functional leadership. Every question needs a "why they'd ask this" rationale tied to the JD or company context. If `pf_story_bank` is empty, flag it and suggest story topics to prepare.

**Citations:** Each question sourced to the JD requirement or company value it probes. Matched stories sourced to `pf_story_bank`. Source type: `manual_entry` (stories) + `job_board` (JD requirements) + `ai_generated` (question prediction).

### 3.1 Degraded Mode (Opaque Recruiter Outreach)

When a role is tracked with partial information — unknown company, unknown role, or both (see main PRD Section 7.1.7) — the Research Brief operates in degraded mode. Each section either generates normally, generates with reduced scope, or is blocked entirely with a message explaining what's needed to unlock it.

#### Company-Unknown Mode (`confidential.company = true`)

The company name is not disclosed. The user may have hints (industry, funding stage, headcount range) stored in `company.knownAttributes`.

| # | Section | Status | Behavior |
|---|---------|--------|----------|
| 1 | Role Decode | ✅ Available | Generates from JD text. Company name replaced with "[Target Company]". |
| 2 | Company Now | ❌ Blocked | "Company not yet disclosed. This section unlocks when you reveal the company." |
| 3 | Funding & Corporate | ❌ Blocked | Same. If `knownAttributes` has funding hints, shows them as "What you know so far: Series D, 200-500 employees." |
| 4 | Competitive Landscape | ⚠ Partial | If JD describes the product area, generates landscape for that area without naming the specific company. "Companies in the [AI developer tools] space include..." |
| 5 | Team & Org Intelligence | ⚠ Partial | Interviewer research works if names provided. Org structure blocked. |
| 6 | Network & Connections | ❌ Blocked | Can't cross-reference without company name. Shows: "Reveal the company to see your connections there." |
| 7 | Fit Analysis | ✅ Available | Maps your bullets against JD requirements — doesn't need company name. |
| 8 | Comp Intelligence | ⚠ Partial | Can estimate range from level + industry + location hints. Less precise without company name. Flagged as approximate. |
| 9 | Strategic Challenges | ⚠ Partial | Can infer from JD language and industry hints. Generic challenges clearly labeled as such. |
| 10 | Culture & Values | ❌ Blocked | "Company not yet disclosed. This section unlocks when you reveal the company." |
| 11 | Questions to Ask | ⚠ Partial | Generates role-specific questions (from JD) but not company-specific ones. Adds: "Once the company is revealed, this section will add company-specific questions." |
| 12 | TMAY Script | ✅ Available | Built from your resume positioned for the role. Uses "[Target Company]" as placeholder. |
| 13 | Likely Interview Qs | ✅ Available | Behavioral + technical derived from JD. Doesn't need company name. |

**Summary:** 5 sections fully available, 4 partial, 4 blocked. The user can still meaningfully prepare for the role.

## 4. MCP Tool: `pf_generate_brief_section`

### Tool Specification

```
Tool: pf_generate_brief_section
Description: Generate one section of a Research Brief using Claude.
             Reads role/company data from localStorage (via bridge) and
             artifacts from MCP storage. Returns generated HTML content
             with inline citation markers.

Input Schema:
  roleId       (string, required)  — Role ID from pf_roles
  companyName   (string, required)  — Company name from pf_companies
  sectionNum   (integer, required)  — Section number (1-13)
  forceRefresh (boolean, optional)  — Ignore cached version, regenerate. Default: false

Output:
  {
    sectionNum: number,
    title: string,
    content: string,          // HTML content with [n] citation markers

```

## 5. Caching & Invalidation

### Cache Strategy

Each section is cached as an artifact in the MCP server (type: `research_brief`, tagged with `{company, roleId, sectionNum}`). The browser module reads from these artifacts.

### Automatic Invalidation Triggers

| Trigger | Sections Invalidated | Reason |
|---------|---------------------|--------|
| JD text changes | All 13 | JD is the anchor — everything depends on it |
| Role title changes | 1, 5, 7, 9, 12 | Level signals and fit mapping change |
| Positioning changes (IC ↔ management) | 1, 5, 7, 9, 12, 13 | Fundamentally different prep |
| Company profile updated | 2, 3, 4, 10 | Company-centric sections need refresh |
| Interviewer names added/changed | 5 | Interviewer profiles section |
| Bullet bank updated | 7, 12, 13 | Fit analysis and scripts use bullets |
| Story bank updated | 13 | Interview questions match stories |
| Connections updated | 6 | Network cross-reference changes |
| Comp cache updated | 8 | Compensation data refreshed |
| Stage changes (e.g., Applied → Screen) | 11 | Questions to ask shift by interview phase |

When a trigger fires, affected sections are marked `stale` in the MCP index (not deleted). The browser shows a "⚠ Stale — data has changed since this was generated" badge with a "Refresh" button. The user decides when to regenerate.

### Manual Controls

- **Refresh single section** — Regenerate one section, preserving others
- **Refresh all** — Regenerate all 13 sections from scratch
- **Clear brief** — Delete all cached sections for this role (soft delete in MCP)

## 7. Export & Artifacts

### Auto-Save

When all 13 sections are generated (no section in `Generating` state), the full brief is automatically compiled into a single HTML artifact and saved to the MCP server:

```
Type: research_brief
Filename: research-brief_{company}_{roleTitle}_{date}.html
Tags: [company, roleId, "complete", date]
Metadata: { sectionsGenerated: 13, inputsMissing: [...], generatedAt: ISO }
```

Previous versions are never overwritten — each generation creates a new artifact. This supports version comparison: "How did my prep evolve between the recruiter screen and the onsite?"

### Export Formats

- **HTML** (default) — Standalone file with embedded CSS, all sections, citations, and source links
- **Print** — Browser print dialog with print-optimized CSS (no sidebars, clean headers)
- **Markdown** — For pasting into notes apps or sharing with mentors

## 10. Success Criteria

A v2 Research Brief is successful when:

1. **Every section names the company, the role, and at least one specific fact** — no generic output that could apply to any company.
2. **Fit Analysis maps your actual bullets against JD requirements** — not platitudes.
3. **Competitive Landscape names real competitors** — not "other companies in the space."
4. **Every factual claim has a citation** — with source type, link, and date.
5. **The brief is useful enough that you'd actually read it before an interview** — the bar is: does this save you 2+ hours of manual research?
6. **Missing data is called out, not worked around** — if we don't have your resume bullets, we say so.
